# ADR-003 — Database Frozen

| Item | Value |
|------|-------|
| ADR ID | ADR-003 |
| Title | Freeze Database Schema |
| Status | Accepted |
| Date | 2026-07-19 |
| Owner | FactoryOS Architecture |
| Decision Type | Architecture |

---

# 1. Context

FactoryOS is built using Google Apps Script (GAS) with Google Sheets as the primary database.

All business logic, repositories, services, and HTML pages depend on a common database schema.

Changing column names, sheet names, or data structures after implementation would introduce significant maintenance costs and increase the risk of regressions.

To ensure long-term stability and predictable development, the database structure must be frozen before Sprint-02 implementation.

---

# 2. Decision

The FactoryOS database schema is officially frozen.

Beginning with Sprint-02:

- No sheet names may be changed.
- No column names may be changed.
- No column order may be changed.
- No existing columns may be deleted.
- Repository interfaces must remain compatible with this schema.

Any future database modification requires:

1. Business review
2. Architecture review
3. New ADR approval
4. Version update

No direct modifications are permitted without this process.

---

# 3. Scope

This ADR applies to:

- Google Sheets database
- Repository layer
- Service layer
- HTML forms
- GAS scripts
- Future modules

---

# 4. Official Database

The official database consists of the following sheets.

| Sheet | Purpose |
|--------|---------|
| PROJECT | Project management |
| TASK | Task management |
| INBOX | Quick capture |
| KNOWLEDGE | Knowledge management |
| SETTINGS | System configuration |

No additional system sheets may be introduced without approval.

---

# 5. Sheet Naming Rules

Rules:

- Sheet names are uppercase.
- English only.
- Singular business concept.
- No spaces.
- No Japanese names.
- No Vietnamese names.

Example:

```
PROJECT
TASK
INBOX
KNOWLEDGE
SETTINGS
```

---

# 6. Column Naming Rules

Rules:

- Uppercase
- Snake Case
- English only
- Stable identifiers
- Never renamed after release

Example:

```
PROJECT_ID
PROJECT_NAME
STATUS
OWNER
CREATED_AT
UPDATED_AT
```

---

# 7. Primary Key Policy

Every master sheet shall contain a unique identifier.

Examples:

| Sheet | Primary Key |
|--------|-------------|
| PROJECT | PROJECT_ID |
| TASK | TASK_ID |
| INBOX | INBOX_ID |
| KNOWLEDGE | KNOWLEDGE_ID |

Primary keys:

- immutable
- unique
- generated by IdService
- never edited manually

---

# 8. SETTINGS Contract

The SETTINGS sheet is the only source of configuration values.

Typical keys include:

```
PROJECT_PREFIX
PROJECT_CURRENT_YEAR
CURRENT_PROJECT_NO

TASK_PREFIX
CURRENT_TASK_NO

INBOX_PREFIX
CURRENT_INBOX_NO

KNOWLEDGE_PREFIX
CURRENT_KNOWLEDGE_NO

DRIVE_ROOT_FOLDER_ID
TEMPLATE_FOLDER_ID

TIME_ZONE
```

Configuration values must never be hardcoded in source code.

---

# 9. Repository Responsibilities

Repositories may:

- Read
- Insert
- Update
- Search

Repositories must not:

- Generate IDs
- Perform business validation
- Implement business rules
- Format UI data

Repositories are responsible only for persistence.

---

# 10. Service Responsibilities

Services are responsible for:

- Validation
- Business rules
- Transaction flow
- ID generation
- Repository orchestration
- Logging

Services must never directly manipulate sheet structures.

---

# 11. HTML Responsibilities

HTML pages:

- Display data
- Collect user input
- Perform client-side validation where appropriate
- Call GAS services

HTML pages must not:

- Access Sheets directly
- Contain business logic
- Generate IDs

---

# 12. Configuration Policy

All configurable values must be stored in SETTINGS.

Examples:

- prefixes
- folder IDs
- timezone
- default values
- numbering sequences

Hardcoded constants are prohibited unless they are true compile-time constants.

---

# 13. Versioning

Database Version:

```
v1.0
```

Version changes occur only when:

- new approved table
- new approved column
- breaking schema change

Each version change requires a new ADR.

---

# 14. Migration Policy

Schema migration is controlled.

Process:

1. Create ADR
2. Review impact
3. Update documentation
4. Implement migration
5. Validate data
6. Release

Direct edits to production Sheets are prohibited.

---

# 15. Risks

Changing the schema without governance may result in:

- broken repositories
- broken services
- invalid HTML forms
- corrupted IDs
- inconsistent reports
- deployment failures

The frozen schema minimizes these risks.

---

# 16. Consequences

Positive:

- Stable architecture
- Predictable development
- Easier maintenance
- Lower regression risk
- Reliable code generation with Codex

Trade-offs:

- Reduced flexibility for ad hoc schema changes
- Formal approval required for database evolution

These trade-offs are accepted to prioritize long-term maintainability.

---

# 17. Compliance Rules

Every Pull Request must satisfy the following:

- No schema modification
- No renamed sheets
- No renamed columns
- No hardcoded configuration
- No direct Sheet access outside Repository layer

PRs violating these rules shall be rejected.

---

# 18. Future Changes

Any future database evolution must include:

- New ADR
- Updated SRS
- Updated TDS
- Migration plan
- Compatibility assessment
- Version increment

Without these artifacts, the database schema remains frozen.

---

# Approval

Status: **ACCEPTED**

This ADR establishes the official and frozen database contract for FactoryOS beginning with Sprint-02.
